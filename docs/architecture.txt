Echo Architecture
=================

Echo is a PHP-compatible compiled language built around LLVM.

Echo has one compiler backend: LLVM.
Echo has two LLVM execution modes: AOT and JIT.

Short Form
----------

  Source -> AST -> HIR -> MIR -> LLVM IR -> AOT/JIT


LLVM-First Execution Model
--------------------------

Echo's primary compiler path is intentionally linear:

  Source -> AST -> HIR -> MIR -> LLVM IR

HIR owns resolved PHP/Echo meaning. MIR is the required compiler IR between HIR
and LLVM. MIR is backend-neutral, but it is not VM bytecode and not a second
execution engine.

LLVM IR is the single backend target. From LLVM IR, Echo can support multiple
execution modes without introducing another language engine:

  LLVM IR -> AOT/native build
  LLVM IR -> JIT/in-process execution

`xo build` uses the AOT path and links generated code with `echo_runtime`.

`xo run` may initially build and run a temporary native binary. Later, it should
move to LLVM JIT for faster development loops.

Embedded Echo should use LLVM JIT in the host process. The host registers
`echo_runtime` symbols so generated LLVM code can call the same runtime
functions used by AOT builds.

This avoids semantic drift because Echo has one compiler backend and one runtime
contract.


Runtime Contract
----------------

`echo_runtime` remains the shared semantic/runtime layer. LLVM-generated code
should call runtime functions such as:

  echo_write
  echo_write_i64
  echo_write_string
  echo_value_add
  echo_to_printable
  echo_array_get
  echo_array_set
  echo_call_dynamic
  echo_ob_start
  echo_ob_flush
  echo_throw
  echo_shutdown

AOT and JIT both use the same runtime contract:

  AOT:
    LLVM IR -> object/binary -> linked with echo_runtime

  JIT:
    LLVM IR -> in-process execution -> registered echo_runtime symbols


LLVM IR as the Execution Boundary
---------------------------------

Echo does not plan a custom bytecode VM or interpreter as a second execution
engine. LLVM IR is the execution boundary for both native builds and in-process
LLVM JIT execution.

For example, the same Echo program could accidentally behave differently if a
custom bytecode engine and LLVM codegen both independently implemented language
behavior.

LLVM JIT gives Echo most of the near-term benefits of embedded execution while
preserving the same LLVM backend used for native builds.

This means portable bytecode, a tiny interpreter, deterministic bytecode
debugger stepping, and plugin scripts without native code execution are outside
the planned architecture unless Echo deliberately revisits the execution
boundary.


Single Language Rule
--------------------

There is one parser configuration, not per-extension parsers.

  *.php
  *.echo
  *.xo

All compile as the same Echo language. File extensions guide ecosystem
conventions and tooling, not syntax or semantic validity.

The AST records source-level facts when later tools need them. It does not lower
syntax into runtime or LLVM concepts.


Ownership Boundaries
--------------------

  echo_source
      Owns source file classification and source text metadata.

  echo_lexer
      Owns tokenization and lexer diagnostics.

  echo_parser
      Owns parsing and parser diagnostics. It should remain one shared parser
      for PHP-compatible syntax plus Echo extensions.

  echo_ast
      Owns syntax tree shape. AST nodes preserve source syntax and source-level
      facts.

  echo_hir
      Owns the first compiler-friendly representation after parsing. HIR is
      derived from AST plus shared semantic facts and preserves enough source
      structure for diagnostics, tooling, and language-level reasoning.

  echo_mir
      Owns the required backend-neutral executable lowering layer between HIR
      and LLVM IR. MIR exists to make code generation regular and testable; it
      is not VM bytecode, an interpreter format, or a second semantic engine.

  echo_semantics
      Owns meaning that requires analysis: variable bindings, expression types,
      scope rules, symbol resolution, and semantic diagnostics.

  echo_codegen
      Owns MIR to LLVM IR generation and ABI routing through inkwell. LLVM is
      the single compiler backend target. Codegen may select stable runtime
      symbols, but it does not own built-in, intrinsic, dynamic-call,
      collection, reflection, or value semantics.

  echo_runtime
      Owns executable value and runtime behavior in Rust: PHP/Echo values,
      collections, output buffering, reflection dispatch, dynamic calls,
      built-ins, standard-library intrinsics, process/task behavior, and other
      observable runtime operations. AOT links it; JIT registers its symbols in
      the host process.

  xo
      Owns CLI orchestration, command presentation, REPL hosting, and display
      formatting. It must not define private language semantics.


MIR Requirement
---------------

MIR is required for the architecture. It is Echo's backend-neutral compiler IR
between HIR and LLVM IR. MIR should make code generation simpler to test and
less tied to AST or HIR details.

MIR must not become a second executable representation or a second semantic
engine. It lowers to LLVM IR; AOT and JIT both execute through LLVM.


Roadmap
-------

Current:

  AST -> LLVM IR -> temp native binary

Next:

  AST -> HIR -> MIR -> LLVM IR -> native binary

Then:

  AST -> HIR -> MIR -> LLVM IR -> JIT


Future Shape
------------

The target flow is:

  source text
    -> shared lexer/parser
    -> AST
    -> HIR
    -> MIR
    -> LLVM IR
    -> AOT/native build or LLVM JIT
    -> Rust-owned runtime behavior

The REPL, embedding hosts, and native builds should consume the same LLVM-first
language pipeline instead of maintaining local parsers, evaluators, symbol
tables, type environments, or execution engines.
